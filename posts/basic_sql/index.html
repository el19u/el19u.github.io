bye bug 是個好東西

Posted on:2022年7月21日 |  at 上午12:27

專案開發階段，偶爾會碰到程式跑到哪裡，只知道出來的東西不符合預期，看 Rails 噴的紅字也沒辦法知道確切的錯誤，關鍵字不夠，拿去 Google 也毫無頭緒，這時 byebug 就是拿來尋找方向的好東西了。

解決 Bug 的方法

要找 Bug 的方法有幾種：

  • 肉眼看 Code
  • puts 大法
  • byebug

撇除用肉眼看 Code，puts 大法在某些時候還滿好用的。

puts "*"*10 #**********
puts "Hello" # Hello
puts "*"*10 #**********

像這樣子就能將想要的內容包在裡面，把錯誤框起來。

不過缺點就是，東西一多就很難框，每次刪除也需要花時間，打的字數也多。

byebug 基本使用方法

Ruby 裡面想要使用 byebug 需要安裝 byebug gem，而 Rails new 起來時已經放在 gem 裡面，不需要額外安裝。

想使用 byebug，只需要在 code 之間寫 byebug 就好。

def process
  byebug
  Post.create(
    title: mail.subject,
    body: body,
    attachments: attachments.map{ |a| a[:blob] }
    )
end

當程式運作到 byebug 時就會自動停下，並在 console 裡面出現(byebug)字樣， 在這時就能下一些指令，開始找 bug 出在哪裡。

byebug 指令

在 byebug 裡面輸入 help，能看見指令表。

其中幾個我自己會用到的是

break

設中斷點，要求 byebug 在抵達該行前停下（byebug）

list

顯示程式執行到哪邊，會輸出附近幾行

next

執行到下一段

continue

繼續執行，或執行到下一個中斷點

step

進入找被呼叫的方法(method)。 用來找 method_missing 錯誤很方便

where

找出方法(method)是被誰呼叫的。 會顯示 call stack 數量，0 是目前被呼叫的 method，數字大的會呼叫數字小的。

display expr

在抵達中斷點時執行 expr

set linetrace

自動執行 expr 監控每一行經過的 code

stop linetrace

停止自動執行 expr

quit

離開 byebug，rails server 也會跟著關閉。

結論

byebug 是個好東西，在寫專案的過程遇到諸多 bug 都是靠它找到問題。 專案中也時常用來確認目前變數裡面的值，善用 byebug，更能找到 google 方向。
