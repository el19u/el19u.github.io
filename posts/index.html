    Email收發信的各種協定

    爲了復刻團隊專案 Gmail，首先需要了解 Email 收發信是怎麼運作，確保自己做的方向正確，一步一步拼湊所需的知識，以及需要準備的哪些東西。 寄信＆收信 SMTP 協定（Simple Mail Transfer Protocol） SMTP 用於驗證 Email 的傳輸。 當使用者寫完信按下送出時，就會透過 SMTP 協定，傳送到 SMTP Server，再發送到收件人的 SMTP Server，等待對方用 POP3 或 IMAP 的方式存取信件。 SMTP 使用 TCP 協定，確保寄信正常。假如因為某些原因，像是拼錯 Email 信箱，或是沒有這個 Email 信箱，自己就會收到寄信失敗的信件。 簡單來說 SMTP 就像郵差，負責運送郵件，假如找不到就會退信，如果找到就會放在收信人的信箱裡。 IMAP （Internet Message Access Protocol） 使用 IMAP 的話，資料會保存在 Server 上。當使用者開啟信箱時，會連線到 Server 確認有無新信件，如果使用者建立了一個標籤或是資料夾，Server 也會同步更新，之後傳到每個裝置上，保持一制性。意思是使用者用什麼裝置做什麼，其他裝置也會更著改變。 簡單來說，IMAP 是讓使用者到伺服器去查看信件，也因此只要有什麼動作發生，就會記錄在伺服器中，當然使用什麼裝置，做什麼變更，都會同步。 POP3 協定（Post Office Protocol） 相較於 IMAP，POP3 單純許多。 只要信件從 Server 進入信箱，就會將備份從 Server 裡刪除。 舉例來說，如果是使用 A 電腦開啟信箱，將信件下載回來，那麼再打開 B 電腦開啟信箱，就不會收到信了，因為信件只會保存在 A 電腦上。...

    Ruby 物件導向設計實踐 敏捷開發入門

    Practical Object-Oriented Design in Ruby: An Agile Primer 本書是 Sandi Metz 大神的著作。 此系列文用來記錄目前的讀書狀況。釐清觀念及深度了解物件導向，學習設計具重複使用、簡單更改特性的物件設計。 物件導向基礎 物件導向有三個特性，目的是為了重複使用，容易修改。 容易修改指的是： 修改後也不會影響其他功能運作 遇到需求變更也只需要小幅更動 封裝（Encapsulation） class Greeting def say_hello(word) puts "#{word}" end end 封裝是將方法包裝起來，像是介面的概念，如同遙控器裡的按鈕，或是 BMI 計算等功能，將其包裝起來。 繼承（Inheritance） class People < Greeting end People 繼承至 Greeting，提供一個可操作介面，而非直接觸碰 Greeting 裡面的方法。 原因在於，直接修改 Greeting 未來新增其他方法後，可能會發生未知的錯誤，並使 Greeting 裡面的東西變得過於龐大，難以修改，也與該物件名稱的所需的行為脫軌。 多型別（Polymorphism） John = People.new("我就棒 123") John.say_hello #我就棒 123 Ellen = People.new("我最棒 456") John.say_hello #我最棒 456 定義相同的物件，產生出不同的效果（型別），就像 John 是我就棒 123，而 Ellen 是 我最棒 456。 總結 上面舉例的方法或許有點抽象，但以 Rails 的方向去思考，從 ExampleController < ApplicationController 開始，就是在做一個繼承的動作，也因此在 Controller 裡面不需要額外多寫什麼 Action 也能正常作用。
